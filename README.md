# Provenance Guard

A Flask backend API that classifies submitted text as AI-generated or human-written, returning a structured JSON response with a confidence score, individual signal scores, and a plain-language transparency label. It also handles creator appeals and maintains a full SQLite audit log.

## 1. Project Overview

Provenance Guard addresses a growing need on creative writing platforms: readers and platforms need a way to know whether posted content was written by a person or generated by an AI tool, without weaponizing that information unfairly against creators. The system is designed around two core commitments:

- **Transparency over verdicts.** The system returns plain-language labels that describe evidence, not verdicts. It acknowledges uncertainty explicitly rather than forcing a binary outcome.
- **False positives cost more than false negatives.** Mislabeling a human's original work as AI-generated harms creators. The system is intentionally conservative: it requires high confidence before issuing an AI label, and provides an appeals pathway for any creator who disagrees with a classification.

The system does not automate consequences. It classifies, logs, and surfaces information for human review.

## 2. Architecture Overview

A submitted text travels through five components before a response is returned.

A POST request to `/submit` arrives at `app.py`, which validates that both `text` and `creator_id` are present and generates a UUID for the submission. The text is then passed simultaneously to two detection pipelines: `pipeline/groq_signal.py` calls the Groq API (model: `llama-3.3-70b-versatile`) and returns a float representing how AI-like the text reads holistically; `pipeline/stylometric.py` computes three structural statistics about the text and returns an independent float. Both scores are passed to `pipeline/combiner.py`, which applies a weighted formula to produce a single confidence score. That score is passed to `labels.py`, which maps it to one of three tiers and returns the corresponding plain-language label text. Finally, `models/audit.py` writes all data — content ID, creator ID, both signal scores, confidence, attribution, status, and timestamp — to a SQLite database (`provenance.db`) before the JSON response is returned to the client.

Appeals arrive at `POST /appeal`. `app.py` validates the request, `models/audit.py` confirms the `content_id` exists, and `models/appeal.py` updates the row's status to `under_review` and records the creator's reasoning. `GET /log` exposes all entries as a structured JSON array.

## 3. Detection Signals

### Signal 1: Groq LLM Classification (`pipeline/groq_signal.py`)

**What it measures:** Whether a piece of writing reads as AI-generated based on holistic semantic and stylistic patterns. The model (`llama-3.3-70b-versatile`) is prompted to rate the text 0–10 on likelihood of being AI-generated, looking specifically for: uniform register throughout, hedged corporate phrasing ("it is important to note that"), parallel sentence constructions, absence of personal voice, and predictable rhetorical flow. The integer score is normalized to a float in [0.0, 1.0].

**Why this property differs between human and AI writing:** AI language models are trained to maximize coherence and predictability. This produces writing that is stylistically consistent across a passage but lacks the irregular, emotionally-driven, voice-specific quality of human expression. Humans write with personal history, idiosyncratic phrasing, and inconsistency that AI models do not naturally replicate. A holistic reader — human or LLM — can often detect this difference even when no single sentence is a giveaway.

**Weight:** 0.6 — semantic holistic judgment is more reliable across content types than structural statistics.

**Blind spots:**
- Cannot detect lightly-edited AI text where a human's edits break the AI's stylistic uniformity just enough
- May misclassify non-native English speakers who write formally and carefully — their writing can exhibit the hedged, uniform register the model associates with AI
- Short texts under 30 words give the model insufficient context for a reliable judgment
- Intentionally formal human writing (speeches, cover letters, eulogies) may score as AI-like

---

### Signal 2: Stylometric Heuristics (`pipeline/stylometric.py`)

**What it measures:** Three structural and statistical properties of the text that differ systematically between human and AI writing, independent of the text's meaning:

**Sub-signal 1 — Sentence Length Variance (SLV):** Computes the standard deviation of sentence lengths in words. AI writing tends to use more uniform sentence lengths; human writing mixes short punchy sentences with longer complex ones. Low variance → higher AI score. Formula: `1.0 - min(std_dev / 12.0, 1.0)`.

**Sub-signal 2 — Type-Token Ratio (TTR):** `unique_word_count / total_word_count`. AI writing recycles vocabulary more predictably; humans introduce more variety, colloquialisms, and domain-specific terms across a passage. Low TTR → higher AI score. Formula: `max(0.0, min(1.0, (0.90 - ttr) / 0.50))`.

**Sub-signal 3 — Punctuation Density (PD):** Counts expressive punctuation marks per word (!, ?, ;, :, parentheses, em-dashes, ellipses — excluding plain periods and commas). Humans use expressive punctuation more liberally to reflect emotional register and personal voice. AI writing uses it sparingly and predictably. Low expressive density → higher AI score.

Final stylometric score = average of the three normalized sub-signals.

**Why these properties differ:** These are not learned associations — they are structural consequences of how AI text is generated. Transformer-based language models produce text by predicting the next token; this tends to produce smooth, medium-length sentences with moderately diverse vocabulary but few emotional punctuation departures. Human writing, by contrast, is shaped by cognitive load, emotional state, and personal habit — all of which introduce structural irregularity.

**Weight:** 0.4 — supplementary structural evidence, useful precisely because it is independent of the LLM signal.

**Blind spots:**
- Academic and legal writing by humans is highly uniform by convention — both signals will flag it as AI-like even when it is not
- Short poems and lyrics: deliberate repetition lowers TTR artificially; fragmented lines make sentence length variance unreliable; minimal punctuation raises the PD score. Texts under 50 words produce statistically unreliable variance measurements
- The TTR sub-signal is unreliable on texts with fewer than 10 words

## 4. Confidence Scoring

### Formula

```
final_confidence = (0.6 × groq_score) + (0.4 × stylometric_score)
```

Both inputs are clamped to [0.0, 1.0] before combination. The result is a float in [0.0, 1.0] where 1.0 represents maximum confidence that the text is AI-generated.

### Weight Rationale

The Groq signal receives a higher weight (0.6) because holistic semantic judgment — the overall "feel" of a piece of writing — is more reliable across content types than structural statistics. The stylometric signal (0.4) is structurally independent of the LLM signal: it captures what the text *looks like* numerically, not what it *reads like* semantically. When both signals agree, confidence is justified. When they disagree, the score lands in the uncertain band — which is the honest outcome when evidence is mixed.

### Asymmetric Threshold Design

| Confidence | Attribution | Label Tier |
|------------|-------------|------------|
| > 0.75 | `likely_ai` | High-confidence AI |
| 0.40 – 0.75 | `uncertain` | Uncertain |
| < 0.40 | `likely_human` | High-confidence Human |

The thresholds are intentionally asymmetric. The AI label requires confidence above 0.75 — not 0.5 — before being applied. This is because a false positive (labeling a human's original work as AI-generated) is significantly worse than a false negative on a creative platform. The 0.75 threshold consciously accepts more false negatives in order to reduce false positives. The appeals workflow exists precisely for the edge cases where the system does issue a label a creator disagrees with.

The lower boundary for the human label (0.40) means the uncertain band is wider than either confident tier. This reflects honest epistemic humility: mixed or moderate signal does not deserve a confident label in either direction.

### Example Submissions

The following scores are real outputs from the live system:

```
Example 1 (high-confidence AI):
  Text: "Artificial intelligence represents a transformative paradigm shift in modern 
         society. It is important to note that while the benefits of AI are numerous, 
         it is equally essential to consider the ethical implications..."
  groq_score=0.90, stylometric_score=0.55, confidence=0.76 → attribution: likely_ai

Example 2 (lower-confidence, human):
  Text: "ok so i finally tried that new ramen place downtown and honestly? 
         underwhelming. the broth was fine but they put WAY too much sodium in 
         it and i was thirsty for like three hours after..."
  groq_score=0.20, stylometric_score=0.48, confidence=0.31 → attribution: likely_human
```

The two examples produce noticeably different confidence scores (0.76 vs. 0.31) and land in different label tiers. The Groq signal is the primary driver of this difference: it rated the formal AI-style text at 0.90 and the casual human text at 0.20. The stylometric signal is more moderate in both cases — the casual human text still has relatively low punctuation density and moderate sentence variance, explaining its 0.48 stylometric score.

### Validation

Scores were validated by testing four deliberate inputs during development:
- Clearly AI corporate text → confidence 0.82 → `likely_ai` (expected)
- Clearly casual human text → confidence 0.19 → `likely_human` (expected)
- Formal academic human text → confidence 0.58 → `uncertain` (correctly not confident enough for AI label)
- Lightly-edited AI output → confidence 0.61 → `uncertain` (correctly uncertain when edits reduce AI markers)

The validation confirmed the thresholds are meaningful: inputs that should clearly land in one tier do so, and inputs with mixed characteristics correctly land in the uncertain band.

## 5. Transparency Labels

The labels are shown to the reader of a platform post, not to a technical reviewer. They use no jargon — no mention of "classifiers," "scores," "probabilities," or "models." All three label variants are written out below exactly as they appear in `labels.py`.

---

**High-confidence AI (confidence > 0.75) — `likely_ai`:**

> "This content was likely generated by AI. Our analysis found patterns strongly associated with AI-generated writing, including uniform sentence structure and consistent vocabulary use. If you created this yourself, you can submit an appeal."

---

**Uncertain (confidence 0.40–0.75) — `uncertain`:**

> "We're not sure whether this was written by a person or an AI. Some patterns in this content are consistent with AI-generated writing, but we don't have enough confidence to make a definitive call. Creators who wrote this themselves can submit an appeal."

---

**High-confidence Human (confidence < 0.40) — `likely_human`:**

> "This content appears to have been written by a person. Our analysis found the kind of variation in style, vocabulary, and structure that's typical of human writing."

---

### Design Choices

**Plain language throughout.** A non-technical reader could understand any of these labels without knowing what a classifier is. The labels describe what was found ("patterns... including uniform sentence structure") rather than what tool was used.

**Evidence, not verdict.** The AI label says "patterns strongly *associated with* AI-generated writing" — it describes evidence, not a definitive finding. The uncertain label explicitly names the system's uncertainty rather than hiding it behind neutral phrasing.

**Appeal pathway in the AI and uncertain labels; not in the human label.** Both the AI and uncertain labels include an invitation to submit an appeal, because both represent classification outcomes a creator might disagree with. The human label is a positive outcome that does not require a remedy — so no appeal invitation appears there.

**The high-AI and high-human labels are substantively different in wording.** The AI label names specific patterns found. The human label names the *type* of variation that supports a human classification. They are not mirror images of each other, and neither contains a percentage or score.

## 6. Appeals Workflow

Any creator who receives an AI or uncertain label and believes the classification is wrong can submit an appeal using the `content_id` from their submission response.

### Submitting an Appeal

```bash
curl -X POST http://localhost:5001/appeal \
  -H "Content-Type: application/json" \
  -d '{
    "content_id": "e1ce53f5-7ab3-4b8c-94e3-ad23972492f2",
    "creator_reasoning": "I wrote this myself as part of a formal business report. I am a non-native English speaker and tend to write formally."
  }'
```

**Response:**
```json
{
  "message": "Appeal received. Your content is now under review.",
  "content_id": "e1ce53f5-7ab3-4b8c-94e3-ad23972492f2",
  "status": "under_review"
}
```

### What Happens on Appeal

1. `app.py` validates that `content_id` and `creator_reasoning` are both present
2. `models/audit.py` confirms the `content_id` exists in the database (returns 404 if not found)
3. `models/appeal.py` updates the audit log row: `status` changes from `"classified"` to `"under_review"`, and `appeal_reasoning` is populated with the creator's written explanation
4. The full entry — original scores, original attribution, and appeal reasoning — becomes visible in `GET /log`

### What a Human Reviewer Sees

When a reviewer calls `GET /log` and looks at entries with `"status": "under_review"`, they see:
- The original `attribution` and `confidence` score
- Both individual signal scores (`groq_score` and `stylometric_score`)
- The `timestamp` of the original classification
- The creator's written `appeal_reasoning`

This gives the reviewer everything needed for a human judgment: how confident was the system? What did each signal detect? What does the creator claim? Re-classification is not automated — a human reviewer makes the final call. This is intentional: automated re-classification creates an incentive to spam appeals until the system flips.

## 7. Rate Limiting

### Specific Limits

`POST /submit` is rate-limited to **10 requests per minute** and **100 requests per day** per IP address, enforced by Flask-Limiter with an in-memory store.

```python
@limiter.limit("10 per minute;100 per day")
def submit():
    ...
```

### Reasoning

These limits are tied to realistic usage on a writing platform:

- **10 per minute:** A legitimate writer submitting their own work for review does not need more than 10 submissions per minute. Even a prolific editor testing revisions of the same passage would be unlikely to exceed this. The limit stops automated script floods (e.g., a bad actor probing the classifier with thousands of inputs to map its decision boundary) without blocking any genuine creative workflow.
- **100 per day:** A heavy writing day — editing multiple pieces, re-submitting drafts — is unlikely to exceed 100 submissions. This covers all realistic individual usage while limiting sustained automated abuse.

These are not default values: the 10/minute limit was chosen to match typical human interaction cadence, and the 100/day limit was chosen to cover an entire active session without resetting mid-day for any legitimate user.

### Rate Limit in Action

Firing 12 rapid requests shows the 429 responses after the 10th:

```
Request 1:  200
Request 2:  200
Request 3:  200
Request 4:  200
Request 5:  200
Request 6:  200
Request 7:  200
Request 8:  200
Request 9:  200
Request 10: 200
Request 11: 429
Request 12: 429
```

A 429 response means the rate limit was exceeded. The client must wait until the sliding window resets (up to 60 seconds for the per-minute limit) before additional submissions are accepted.

## 9. Known Limitations

### Limitation 1: Non-native English speakers writing formally

This is the most serious false-positive risk for this system. A writer who learned English through academic materials — textbooks, formal correspondence, professional documents — may naturally produce text with uniform sentence structure, hedged phrasing, conservative vocabulary, and minimal expressive punctuation. This is not AI-generated writing; it is writing shaped by formal-register training.

Both signals will incorrectly flag this text:

- **Signal 1 (Groq):** The LLM is explicitly prompted to detect "hedged corporate phrasing," "uniform register," and "absence of personal voice." Formal non-native writing often exhibits all three of these markers. The model has no way to distinguish between "an AI produced this uniform register" and "a careful non-native writer produced this uniform register."
- **Signal 2 (Stylometric):** The TTR sub-signal will detect conservative, formal vocabulary as low variety. The SLV sub-signal will detect careful, measured sentence construction as low variance. The PD sub-signal will detect minimal exclamation points and parenthetical asides as low expressive density. All three sub-signals push the stylometric score toward AI-like even when the text is original human writing.

The appeals workflow is the primary mitigation. A creator in this situation can submit an appeal explaining their writing background, and a human reviewer can examine the signal scores and reasoning to make an informed override.

### Limitation 2: Short poems, haiku, and lyric text (under 50 words)

The stylometric signal's three sub-signals are statistics — they require a meaningful sample size to produce reliable readings. On a 20-word haiku or a 4-line stanza:

- **SLV** is computed from fewer than 5 sentences, making the standard deviation statistically meaningless
- **TTR** is artificially elevated (short texts with no word repetition score as very high lexical variety), or artificially deflated by deliberate repetition in a refrain — in either case the signal is unreliable
- **PD** is dominated by line-break punctuation choices that do not reflect the intended signal

The current implementation does not apply a minimum-length guard; it applies the formula regardless. A 15-word haiku will produce a stylometric score, but that score carries essentially no information. This is a known gap in the implementation.

## 10. Spec Reflection

### One Way the Spec Helped

The CLAUDE.md spec was precise about the asymmetric threshold design before a single line of code was written. The explicit instruction that ">0.75 is required for the AI label" — with written rationale about false-positive costs — meant there was no negotiation during implementation. When testing showed that some formal human text was landing at 0.65–0.72, the system correctly placed it in the uncertain band rather than labeling it AI. The spec's explicitness made this outcome feel correct rather than like a bug.

### One Way Implementation Diverged from the Plan

The stylometric signal's punctuation density sub-signal was specified in the initial planning document as measuring overall punctuation density — all punctuation marks per word (`[^\w\s]`). During implementation, this was changed to measure only *expressive* punctuation marks (!, ?, ;, :, parentheses, em-dashes, ellipses) — explicitly excluding periods and commas.

The reason for the change: periods and commas appear in both AI and human formal writing at roughly similar rates. AI writing uses them grammatically; formal human writing uses them grammatically. Counting them adds noise without adding signal. Worse, it would make formal AI text appear slightly more punctuation-dense than sparse AI text, creating a normalization inversion where the signal partially cancels itself.

By restricting to expressive punctuation — the marks that reflect emotional register and personal voice — the sub-signal captures the intended theoretical property: human writers use punctuation beyond the grammatical minimum; AI writers do not. This refinement was noted in a code comment in `pipeline/stylometric.py`:

> "Note: Using expressive-only punctuation (not all `[^\w\s]`) is the correct normalization because the theoretical basis is expressive register, not total punctuation count. Plain periods/commas appear equally in formal AI output and formal human output and add no discriminative signal."

## 11. AI Usage

### Instance 1: CoreAPIAgent — Flask skeleton and Groq signal (Milestone 3)

**What was directed:** The agent was provided with the full API contract from CLAUDE.md (POST /submit request/response schema, GET /log schema, audit log field requirements) plus the Detection Signal 1 specification (Groq model, prompt design, output normalization, error fallback). The request was to generate the Flask application skeleton with both routes stubbed, the `pipeline/groq_signal.py` implementation, and the `models/audit.py` SQLite layer.

**What the AI produced:** A working Flask app with POST /submit, GET /log, and POST /appeal stubs. The groq_signal.py implementation followed the spec's prompt design (rate 0–10, return JSON, normalize by dividing by 10). The audit.py layer implemented `init_db`, `log_submission`, `get_log`, and `get_entry` with the correct SQLite schema.

**What was revised:** The port was adjusted from 5000 to 5001 because macOS AirPlay Receiver occupies port 5000 by default and the server would fail to bind. The agent's generated code also lacked handling for Groq's occasional markdown code-fence wrapping of JSON responses (the API sometimes returns ` ```json\n{...}\n``` ` instead of raw JSON). A code-fence stripping block was added to the `analyze` function to handle this reliably.

---

### Instance 2: PipelineAgent — Stylometric signal and combiner (Milestone 4)

**What was directed:** The agent was provided with the Detection Signal 2 specification from CLAUDE.md (the three sub-signals: SLV, TTR, PD; each normalized to [0,1]; final score = average), the uncertainty threshold table (0.75 / 0.40 boundaries), the confidence formula, and the architecture diagram. The request was to generate `pipeline/stylometric.py` with all three sub-signals implemented and `pipeline/combiner.py` with the weighted fusion function.

**What the AI produced:** A complete `stylometric.py` with all three sub-signals, correctly normalized, with edge-case handling for short texts (returning 0.5 for insufficient signal). The `combiner.py` was produced with the correct 0.6/0.4 weighting and input clamping.

**What was revised:** The punctuation density sub-signal was initially implemented as the agent produced it — counting all punctuation (`[^\w\s]`). After testing with a formal human text sample, the score was found to be pulling toward AI-like due to comma density in a grammatically complex passage. The metric was revised to count only expressive punctuation marks (!, ?, ;, :, parentheses, em-dashes, ellipses), explicitly excluding periods and commas. This change is documented in a code comment in `stylometric.py`. The agent-generated normalization for TTR was also reviewed: the formula `(0.90 - ttr) / 0.50` assumes a human TTR range of 0.60–0.90 and an AI TTR range of 0.40–0.60; this was verified against the test corpus before accepting it.

## 12. Running the Project

### Setup

```bash
# Create and activate a virtual environment
python3 -m venv .venv
source .venv/bin/activate

# Install dependencies
pip install -r requirements.txt
```

### Environment

Create a file named `groq_key.env` in the project root with:
```
GROQ_API_KEY=your_groq_api_key_here
```

This file is gitignored and must not be committed.

### Start the App

```bash
python3 app.py
```

The server runs on port **5001**.

### Endpoints

| Method | Path | Description |
|--------|------|-------------|
| POST | `/submit` | Submit text for classification. Returns content_id, attribution, confidence, both signal scores, and label text. Rate limited: 10/min, 100/day. |
| POST | `/appeal` | Submit an appeal for a classification. Requires content_id and creator_reasoning. |
| GET | `/log` | Retrieve the full audit log as a structured JSON array. |

### Example Usage

**Submit text for analysis:**
```bash
curl -X POST http://localhost:5001/submit \
  -H "Content-Type: application/json" \
  -d '{"text": "Your text here...", "creator_id": "your-user-id"}'
```

**Submit an appeal:**
```bash
curl -X POST http://localhost:5001/appeal \
  -H "Content-Type: application/json" \
  -d '{
    "content_id": "content-id-from-submit-response",
    "creator_reasoning": "I wrote this myself because..."
  }'
```

**View the audit log:**
```bash
curl http://localhost:5001/log | python3 -m json.tool
```
